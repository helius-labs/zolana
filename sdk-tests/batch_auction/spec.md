# Batch Auction

- No maker and no quoted price: a market is just an asset pair, and the counterparty is chosen per batch by an open English auction on one uniform clearing price. The winning solver fills every order from its own shielded liquidity at settlement -- during the auction the only locked capital is each taker's order and the current best bidder's bond.
- One uniform directed clearing price per batch: every order resolves against the same public `clearing_price`, set once when the bid window closes, so settlement order between fills is irrelevant and no per-order fill leaves a price trace.
- An order's `amount` and `limit_price` (the taker's private minimum clearing price) are never stored or transmitted in the clear; they live only inside the order UTXO's commitment. Bids strictly increase, so the fillable order set is monotone: once the price clears a limit it stays cleared. An order whose limit the final price never reached refunds inside the same `fill` instruction -- identical shape and verifying key -- so an observer cannot tell filled from refunded.
- The program maintains a per-batch order accumulator (`acc <- Poseidon(acc, order_utxo_hash)`, frozen at `join_deadline`) and each order note is additionally encrypted to the batch's operator key. After joining closes, the operator publishes the batch's aggregate volume as a bucket `[lo, hi)` with a proof against the frozen accumulator -- verified by bidders in their own client; the Solana program has no verifier for it. The accumulator makes the disclosure self-authenticating: the operator can withhold the aggregate, but cannot misstate it or show different bidders different values. Buckets rather than the exact sum blunt join-sybil differencing.
- The operator is disclosure-only: it never custodies funds and never executes. It is trusted for liveness twice -- publishing the aggregate to bidders, and forwarding the decrypted order notes to the winner after award (the winner needs each order's witnesses to build fill proofs). The winner therefore learns the batch's composition; losing bidders and the public learn only the bucket.
- Bonds replace reservations: each bid deposits the batch's fixed public `bond` (displacing and refunding the previous bidder's). A winner that fills or refunds every order reclaims it; orders still open past `execution_deadline` are reclaimed by their takers, each reclaim claiming an equal share of the forfeited bond.

## Instructions

| # | Instruction | Tag | Description | Accounts Read | Accounts Modified | Access control |
|---|-------------|-----|-------------|---------------|-------------------|----------------|
| 1 | create_market | 1 | Creates a directed market for a source/destination asset pair. The market account holds only the two asset registry ids and their UTXO commitments (the proof checks the order and fill UTXOs against these commitments). No authority and no price. | — | market account (created) | Permissionless (caller pays rent); the PDA is derived from the asset ids, so each market exists once |
| 2 | open_batch | 2 | Opens a batch on a market. Stores the operator's disclosure encryption key, the fixed `bond` amount, and three slot deadlines (`now <= join_deadline < bid_deadline < execution_deadline`). Initializes `order_accumulator = 0`, `order_count = 0`, `clearing_price = 0`. | market account | batch account (created) | Operator signs (fee payer); anyone may act as operator -- the role carries no custody |
| 3 | place_order | 3 | Creates an order in a batch's join window (`slot <= join_deadline`). One IN1_OUT2 `order_open` proof spends the taker's source UTXO into the order UTXO (owned by the batch's `escrow_authority` PDA, asset bound to the market's source asset) and the taker's change. `amount` and `limit_price` are private witnesses; the proof fixes the payout destination to the source UTXO's owner and commits it into the order UTXO's data hash, so it stays confidential. The program stores the order UTXO hash and `owner`, folds the hash into the batch accumulator, and increments `order_count`. The instruction carries the order note encrypted to the batch's operator key. | market account, batch account | batch account, order account (created) | Taker signs (source UTXO owner, fee payer) |
| 4 | bid | 4 | Improves the batch's uniform clearing price during the bid window (`join_deadline < slot <= bid_deadline`): the bid must strictly exceed the current `clearing_price` and deposits the batch's `bond`, refunding the displaced bidder's bond in the same instruction. Records the bidder as `solver`. Bidding is informed by the operator's off-chain aggregate disclosure, proven against the frozen accumulator. No proof on-chain. | — | batch account | Permissionless: this is the solver competition; strict improvement plus the bond bound what any bidder can do |
| 5 | fill | 5 | Resolves one order -- fill or limit-refund -- during the execution window (`bid_deadline < slot <= execution_deadline`; requires a nonzero `clearing_price`). One IN2_OUT3 `fill` proof spends the order UTXO and the winner's liquidity UTXO (whose asset must be the market's destination asset). If `clearing_price >= limit_price` (private) it fills: the taker's confidential recipient receives `amount * clearing_price` of the destination asset, the winner receives the order's source asset and its liquidity change. Otherwise it refunds: the recipient receives the full `amount` back in the source asset, the winner's liquidity returns whole, and the third output is zero-amount -- identical shape and verifying key, so the outcome is not observable. Closes the order account and decrements `order_count`. | market account, batch account | batch account, order account (closed) | Structurally the winning solver: the liquidity input requires the winner's own UTXO signature, and all destinations are fixed by the proof |
| 6 | reclaim | 6 | Returns an unresolved order to its taker after the winner's window (`slot > execution_deadline`, or `slot > bid_deadline` if no bid landed). One IN1_OUT1 `order_reclaim` proof spends the order UTXO back to the confidential recipient committed in its data hash. If a bond was posted, the reclaimer receives `bond_remaining / order_count` -- the winner's forfeiture for stranding this order. Closes the order account and decrements `order_count`. Reclaim is the visible failure path: unlike fill/refund it is distinguishable. | market account, batch account | batch account, order account (closed) | Permissionless in principle; in practice only the taker holds the order witnesses |
| 7 | close_batch | 7 | Closes a resolved batch (`order_count == 0`, `slot > execution_deadline`): returns the remaining bond to `solver` (whole if every order was filled or refunded) and the batch rent to the operator. | — | batch account (closed) | Permissionless: destinations are fixed by batch state |
